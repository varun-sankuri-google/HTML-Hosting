Reddit Summary

A solo founder experienced a massive, unexpected BigQuery bill of nearly $7,000 after two days of testing a 50GB dbt project. Migrating from AWS Redshift, where the same workload ran for a predictable $400/month, the user was shocked to find their project scanned over 433 tebibytes of data. The core issue stems from a misunderstanding of BigQuery's on-demand, bytes-scanned pricing model compared to Redshift's fixed-cost, provisioned-capacity model. The user's unoptimized dbt queries likely caused repeated full table scans, leading to the astronomical processing volume and subsequent bill, highlighting a critical need for cost-control awareness, such as partitioning and query optimization, when using serverless data warehouses.

Gemini Reasoning

The Reddit thread highlights a user who incurred a massive, unexpected bill while using dbt with BigQuery. The user was unfamiliar with BigQuery's on-demand pricing model and how unoptimized dbt models can lead to scanning terabytes of data. The 'Billing' section of this document should be updated to include a strong cautionary note about this risk, especially for users migrating from fixed-cost data warehouses. It should explain how dbt models can cause full table scans and recommend cost-control best practices like partitioning, clustering, and setting cost controls to prevent bill shock.

712Understanding and Controlling Costs
713 
714Warning: BigQuery's on-demand pricing model charges for the amount of data proce
 >ssed (bytes scanned) by your queries. Users migrating from data warehouses with 
 >fixed-cost or provisioned-capacity pricing models (like AWS Redshift) can be sur
 >prised by large bills if queries are not optimized for a pay-per-scan model. A s
 >ingle unoptimized dbt model can inadvertently scan hundreds of terabytes of data
 >, resulting in significant costs.
715 
716BigQuery's performance is powerful, but this power requires a mindful approach t
 >o cost management, especially during development and testing with tools like dbt
 >. A typical dbt project involves running models repeatedly. If these models perf
 >orm full table scans on large tables, the cumulative data processed can grow exp
 >onentially, leading to unexpected charges.
717 
718To prevent bill shock and use dbt on BigQuery cost-effectively, implement the fo
 >llowing best practices:
719 
720- **Partition and Cluster Your Tables:** This is the most critical cost-control 
 >measure. Partitioning tables by date or another relevant key allows your dbt mod
 >els to scan only the necessary partitions instead of the entire table. Clusterin
 >g further sorts the data within partitions, reducing scan sizes even more. For m
 >ore information, see Introduction to partitioned tables.
721 
722- **Use Materialized Views for Common Aggregations:** If your dbt project involv
 >es complex and repeated aggregations, pre-calculating them with materialized vie
 >ws can drastically reduce the amount of data processed by downstream models.
723 
724- **Set Cost Controls:** Before running any dbt project, configure cost controls
 > at the user and project level. You can set a maximum bytes billed per query to 
 >prevent any single query from running away with costs. See Create custom cost co
 >ntrols for more information.
725 
726- **Monitor Costs and Set Up Alerts:** Actively monitor your BigQuery spending i
 >n the Google Cloud console and set up billing alerts to be notified when costs e
 >xceed a certain threshold. See Monitor your costs and Set budget alerts for more
 > information.
727 
728- **Develop on Data Subsets:** When developing dbt models, use `LIMIT` clauses i
 >n your source definitions or configure dbt to build against smaller sample datas
 >ets to reduce the amount of data scanned during test runs.
